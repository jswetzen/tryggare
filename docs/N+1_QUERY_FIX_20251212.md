# N+1 Query Performance Fix

**Date:** 2025-12-12
**Issue:** Child list endpoint had N+1 query problem
**Status:** Fixed and committed

## Problem Description

The `/api/children/` endpoint was executing 27 database queries for 12 children instead of the expected 4 queries. This was a classic N+1 query problem where each child triggered 2 individual database queries to check their check-in status.

### Query Breakdown (Before Fix)

**Good queries (1-3):**
1. Main query: Fetches children with families (JOIN)
2. Event tickets: Fetches all event tickets for children (IN clause)
3. Session tickets: Fetches all session tickets for children (IN clause)

**Bad queries (4-27):**
- 24 individual queries checking for active check-in records
- 2 duplicate queries per child (12 children × 2 = 24)
- Each query: `SELECT FROM check_in_records WHERE child_id = X AND check_out_time IS NULL`

**Total: 27 queries**

## Root Cause

The `ChildSerializer` has two methods that check check-in status:
- `get_is_checked_in()` - Returns boolean if child is checked in
- `get_active_checkin_id()` - Returns the ID of active check-in record

Both methods had fallback logic to query the database individually when check-in records weren't prefetched. The `ChildViewSet.get_queryset()` was missing the prefetch for check-in records, causing all children to hit the fallback path.

## Solution

### 1. Added Prefetch to ChildViewSet

**File:** `/workspace/check-ins/backend/families/views.py`

```python
def get_queryset(self):
    """
    Optimize queries with select_related and prefetch_related.
    Includes ticket information and check-in status to avoid N+1 queries.
    """
    from checkins.models import CheckInRecord

    event_ticket_prefetch = Prefetch(
        'event_tickets',
        queryset=EventTicket.objects.select_related('event')
    )
    session_ticket_prefetch = Prefetch(
        'session_tickets',
        queryset=SessionTicket.objects.select_related('session', 'session__event')
    )
    # NEW: Prefetch all check-in records to avoid N+1 queries
    checkin_prefetch = Prefetch(
        'checkin_records',
        queryset=CheckInRecord.objects.all()
    )

    return Child.objects.select_related("family").prefetch_related(
        event_ticket_prefetch,
        session_ticket_prefetch,
        checkin_prefetch,  # NEW
    ).all()
```

### 2. Enhanced Serializer to Use Prefetched Data

**File:** `/workspace/check-ins/backend/families/serializers.py`

Updated both `get_is_checked_in()` and `get_active_checkin_id()` methods to:
1. First check for `active_checkins` attribute (used by FamilyViewSet)
2. Then check for prefetched `checkin_records` (used by ChildViewSet)
3. Fall back to database query if neither is available

```python
def get_is_checked_in(self, obj: Child) -> bool:
    # Use prefetched active_checkins if available (Family viewset uses this)
    if hasattr(obj, 'active_checkins'):
        return len(obj.active_checkins) > 0

    # Use prefetched checkin_records if available (Child viewset uses this)
    if hasattr(obj, '_prefetched_objects_cache') and 'checkin_records' in obj._prefetched_objects_cache:
        checkin_records = obj.checkin_records.all()
        return any(record.check_out_time is None for record in checkin_records)

    # Fallback: query if not prefetched
    from checkins.models import CheckInRecord
    active_checkin = CheckInRecord.objects.filter(
        child=obj,
        check_out_time__isnull=True
    ).first()
    return active_checkin is not None
```

**File:** `/workspace/check-ins/backend/families/tests.py`

Updated `test_child_list_query_performance` to expect 4 queries instead of 3:

## Performance Results

### Before Fix
- **Total queries:** 27
- **N+1 queries:** 24 (12 children × 2 duplicate queries each)
- **Efficient queries:** 3

### After Fix
- **Total queries:** 4
- **N+1 queries:** 0
- **Efficient queries:** 4

All queries now use efficient `IN` clauses:
1. `SELECT children ... WHERE child_id IN (...)`
2. `SELECT event_tickets ... WHERE child_id IN (...)`
3. `SELECT session_tickets ... WHERE child_id IN (...)`
4. `SELECT check_in_records ... WHERE child_id IN (...)`

## Lessons Learned

1. **Always prefetch related data** when serializers access relationships
2. **Use Django Debug Toolbar** or `assertNumQueries` to catch N+1 problems early
3. **Check for `_prefetched_objects_cache`** to detect if data is prefetched
4. **Provide fallbacks** in serializers for when data isn't prefetched (but fix the root cause)
5. **Test query counts** explicitly to prevent performance regressions
