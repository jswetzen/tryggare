# Conference Child Management System - Technical Design Document

## Overview
This document outlines the technical architecture and technology stack decisions for the Conference Child Management System. All code will be generated by LLMs, so choices prioritize clarity, type safety, and maintainability.


---

## Technology Stack

### Core Framework
- **Next.js 14+** (App Router)
  - Full-stack React framework
  - Single codebase for frontend and backend
  - File-based routing
  - Server and client components
  - Built-in optimization and performance features

### Language
- **TypeScript**
  - End-to-end type safety
  - Self-documenting code
  - Catches errors at compile time
  - Excellent LLM code generation
  - Better IDE support and autocomplete

### API Layer
- **tRPC**
  - Type-safe API without code generation
  - End-to-end type safety from database to UI
  - No REST/GraphQL boilerplate
  - Automatic type inference
  - Minimal code for API endpoints

### Database
- **PostgreSQL 16**
  - Robust, ACID-compliant
  - Excellent for relational data model
  - Strong audit logging capabilities
  - Good GDPR compliance support
  - No vendor lock-in

### ORM
- **Prisma**
  - Type-safe database client
  - Declarative schema definition
  - Automatic migrations
  - Excellent TypeScript integration
  - Built-in middleware for audit logging
  - Query builder prevents SQL injection

### UI Framework
- **React 18+**
  - Component-based architecture
  - Rich ecosystem
  - Excellent mobile responsiveness
  - Server and client components

### Styling
- **Tailwind CSS**
  - Utility-first CSS
  - Built-in dark mode support
  - CSS variables for theming
  - Minimal custom CSS needed
  - Responsive design utilities

### UI Components
- **shadcn/ui**
  - Copy-paste component library (not npm package)
  - Built on Radix UI primitives
  - Accessible by default
  - Customizable with Tailwind
  - Dark mode built-in

### Internationalization
- **next-intl**
  - Built for Next.js App Router
  - Type-safe translations
  - Automatic locale detection
  - Server and client component support
  - Date/time/number formatting per locale

### Authentication
- **NextAuth.js v5** (Auth.js)
  - Built for Next.js
  - Session management
  - Multiple authentication strategies
  - CSRF protection
  - Type-safe session data

### Package Manager
- **pnpm**
  - Faster than npm/yarn
  - Disk space efficient (symlinks to global store)
  - Strict dependency management (no phantom dependencies)
  - Better monorepo support
  - Industry standard for modern Next.js projects

## Development Workflow

### Initial Setup
1. Initialize project with T3 Stack (Next.js + tRPC + Prisma + Tailwind)
2. Add shadcn/ui components as needed
3. Configure next-intl for internationalization
4. Set up Prisma schema for data model
5. Configure Docker and docker-compose files

### Development Cycle
1. Define data model in Prisma schema
2. Generate Prisma client (auto-updates types)
3. Create tRPC API endpoints (type-safe)
4. Build React components (type-safe props)
5. Style with Tailwind utilities
6. Test in development mode
7. Commit and push to git
8. Portainer auto-deploys

### Database Changes
1. Modify Prisma schema
2. Create migration: `pnpm prisma migrate dev`
3. Migration runs automatically on deployment
4. Types update throughout application

## Technology Decision Rationale

### Why Next.js + TypeScript?
- **Shortest code**: Single codebase, minimal boilerplate
- **Cleanest code**: Type safety, clear conventions
- **Most maintainable**: Self-documenting through types
- **LLM-friendly**: Excellent training data, consistent patterns
- **Fast development**: Hot reload, instant feedback

### Why tRPC?
- **Zero API boilerplate**: No REST endpoints to define
- **Type safety**: End-to-end with no code generation
- **LLM-optimal**: Clean, minimal code output

### Why Prisma?
- **Type-safe queries**: Database errors caught at compile time
- **Migration management**: Schema changes tracked and versioned
- **Audit logging**: Middleware for automatic tracking
- **LLM-friendly**: Declarative schema, clear patterns

### Why pnpm?
- **Faster**: Better performance than npm/yarn
- **Reliable**: Strict dependency resolution
- **Modern**: Industry standard for new projects
- **Compatible**: Nearly identical commands to npm

### Why Portainer GitOps?
- **Simplest deployment**: No separate CI/CD setup
- **Automatic**: Push code → auto-deploy
- **Self-contained**: Everything in one place
- **Visual management**: GUI for stack management
