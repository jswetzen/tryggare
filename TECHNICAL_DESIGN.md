# Conference Child Management System - Technical Design Document

## Overview
This document outlines the technical architecture and technology stack decisions for the Conference Child Management System. The system uses Django for backend/admin and SvelteKit for interactive real-time interfaces. All code will be generated by LLMs, so choices prioritize clarity, simplicity, and maintainability.

---

## Technology Stack

### Backend Framework
- **Django 5.x**
  - Mature Python web framework
  - Built-in admin interface (free CRUD for all models)
  - Excellent ORM with migrations
  - Strong security defaults
  - Well-documented, stable ecosystem
  - Battle-tested Docker deployment

### Real-Time Layer
- **Django Channels 4.x**
  - WebSocket support for Django
  - Mature, first-party solution
  - Channel layers for message broadcasting
  - Integrates seamlessly with Django
  - ASGI server support (Daphne)

### API Layer
- **Django REST Framework (DRF)**
  - RESTful API for Svelte frontend
  - Serializers for data validation
  - Authentication integration
  - Browsable API for development

### Database
- **PostgreSQL 16**
  - Robust, ACID-compliant
  - Excellent for relational data model
  - Strong audit logging capabilities
  - Good GDPR compliance support
  - No vendor lock-in

### Message Broker
- **Valkey (Redis-compatible)**
  - Channel layer for WebSocket broadcasting
  - Message broker between Django workers
  - Drop-in Redis replacement (open source)
  - Simple, reliable, fast

### Frontend Framework
- **SvelteKit**
  - Minimal JavaScript framework
  - File-based routing (like Next.js)
  - Compiles to vanilla JS (no virtual DOM)
  - Small bundle size (~30KB vs React ~150KB)
  - Built-in reactivity (perfect for real-time)
  - SPA mode (static build, no SSR needed)

### Language
- **Python** (Backend)
  - Clear, readable syntax
  - Excellent LLM code generation
  - Mature ecosystem
  - Developer preference

- **TypeScript** (Frontend, optional but recommended)
  - Type safety for Svelte components
  - Better IDE support
  - Catches errors early
  - Can use plain JavaScript if preferred

### Styling
- **Tailwind CSS**
  - Utility-first CSS
  - Built-in dark mode support
  - CSS variables for theming
  - Minimal custom CSS needed
  - Responsive design utilities

### UI Components
- **shadcn-svelte** (optional)
  - Svelte port of shadcn/ui
  - Copy-paste components
  - Accessible by default
  - Customizable with Tailwind

### Internationalization
- **Django i18n** (Backend/Admin)
  - Built-in Django feature
  - Translation files (.po format)
  - Model field translations
  - Date/time/number formatting

- **svelte-i18n** (Frontend)
  - Lightweight i18n for Svelte
  - JSON translation files
  - Reactive locale switching
  - Can share translations with Django via JSON

### Authentication
- **Django Authentication**
  - Built-in user model
  - Session-based authentication
  - CSRF protection
  - Admin-only access control

### Package Manager
- **pip** (Python) + **requirements.txt**
  - Standard Python package management
  - Simple, reliable
  - Docker-friendly

- **pnpm** (JavaScript)
  - Faster than npm/yarn
  - Disk space efficient
  - Strict dependency management
  - Industry standard for modern projects

## Real-Time Architecture

### WebSocket Communication

**Connection Flow:**
1. SvelteKit app connects to Django via WebSocket on page load
2. Django authenticates and adds to broadcast group
3. Client receives real-time updates for session events
4. Connection maintained throughout session
5. Reconnect logic handles temporary disconnections

**Message Types:**
- `child_checked_in` - Child checked into session
- `child_checked_out` - Child checked out from session
- `session_started` - New session activated
- `session_ended` - Session closed
- `emergency_alert` - System-wide notification (future)

**Django Channels Consumer:**
```
CheckInConsumer (WebSocket handler)
  ↓
Receives message from any Django view
  ↓
Broadcasts to "check_in_updates" group
  ↓
All connected clients receive update
```

**Svelte Store Pattern:**
```
WebSocket connection in store
  ↓
onmessage handler updates writable store
  ↓
Components subscribe to store ($checkedInChildren)
  ↓
UI updates automatically (Svelte reactivity)
```

### Why Valkey/Redis?

**Browsers do NOT talk directly to Valkey.** Architecture:

```
Browser ←WebSocket→ Django Channels ←Channel Layer→ Valkey
```

Valkey acts as a **message broker** to coordinate between multiple Django worker processes, enabling broadcasts to all connected WebSocket clients.

## Development Workflow

### Initial Setup

**Backend:**
1. Create Django project
2. Define models (Child, Parent, Session, CheckIn, etc.)
3. Create migrations: `python manage.py makemigrations`
4. Apply migrations: `python manage.py migrate`
5. Create superuser: `python manage.py createsuperuser`
6. Configure Django Channels routing
7. Create DRF serializers and views
8. Create WebSocket consumers

**Frontend:**
1. Initialize SvelteKit: `npm create svelte@latest`
2. Configure for SPA mode (no SSR)
3. Install dependencies (Tailwind, svelte-i18n)
4. Create route structure
5. Build WebSocket store
6. Create UI components

### Development Cycle

1. **Define/modify Django models**
2. **Create migration**: `python manage.py makemigrations`
3. **Run backend**: `python manage.py runserver`
4. **Run frontend**: `npm run dev`
5. **Test in browser**
6. **Commit and push**
7. **Portainer auto-deploys**

### Database Changes

1. Modify Django model
2. Generate migration: `python manage.py makemigrations`
3. Migration runs automatically on deployment
4. Django ORM handles schema changes

## Deployment Architecture

### Containerization

**Django Dockerfile:**
- Multi-stage build (install dependencies separately)
- Run migrations on startup
- Daphne ASGI server for WebSocket support
- Collect static files

**SvelteKit Dockerfile:**
- Build static SPA
- Serve via nginx
- No Node.js runtime needed in production

### Docker Compose

```
services:
  - django (Daphne ASGI server)
  - postgres (database)
  - valkey (message broker)
  - svelte (nginx serving static files)
```

### Portainer GitOps

- **Git repository** contains docker-compose.yml and Dockerfiles
- **Automatic builds** on git push (webhook or polling)
- **Stack management** via Portainer web UI
- **Environment variables** configured in Portainer
- **No external CI/CD** required

### Deployment Flow

```
git push
  ↓
Portainer detects change
  ↓
Pull latest code
  ↓
Build Django image (pip install, collect static)
  ↓
Build Svelte image (npm build, copy to nginx)
  ↓
Run migrations (Django entrypoint)
  ↓
Restart containers
  ↓
Health checks pass
  ↓
Deployment complete
```

## Technology Decision Rationale

### Why Django?
- **Mature, stable framework** - 18+ years of development
- **Built-in admin** - free CRUD interface for all models
- **Excellent ORM** - type-safe queries, migrations
- **Django Channels** - first-party WebSocket support
- **Simple Docker deployment** - battle-tested patterns
- **Python** - developer preference, LLM-friendly
- **Security defaults** - CSRF, XSS, SQL injection protection

### Why Django Channels?
- **Mature real-time solution** - production-ready since 2016
- **Seamless Django integration** - uses same auth, ORM, etc.
- **Well-documented** - extensive guides and examples
- **Proven at scale** - used by major platforms
- **Simple architecture** - easier than custom WebSocket server

### Why SvelteKit over React?
- **30% less code** - more concise than React
- **Smaller bundles** - 5x smaller than React (30KB vs 150KB)
- **True reactivity** - no useState/useEffect complexity
- **Better forms** - bind:value instead of onChange
- **Faster performance** - compiles to vanilla JS, no virtual DOM
- **Simpler for real-time** - reactive stores perfect for WebSocket updates

### Why SvelteKit over Plain Svelte?
- **File-based routing** - clean URL structure
- **Built-in build system** - Vite configured out of the box
- **SPA mode** - static build, no SSR overhead
- **Minimal overhead** - ~50KB additional, not bloated like Next.js
- **Better DX** - hot reload, error messages, dev tools

### Why Valkey over Redis?
- **Open source** - truly free, no licensing concerns
- **Drop-in replacement** - 100% Redis-compatible
- **Community-driven** - not controlled by single vendor
- **Same performance** - identical protocol and features

### Why PostgreSQL?
- **Robust, ACID-compliant** - data integrity guaranteed
- **Excellent Django support** - first-class ORM integration
- **JSON fields** - flexible data when needed
- **Full-text search** - built-in search capabilities
- **No vendor lock-in** - open source, portable

### Why This Stack Overall?
- **Simple deployment** - Django Docker patterns are proven
- **Real-time built-in** - not bolted on as afterthought
- **Python + JavaScript** - two languages total
- **Minimal dependencies** - no complex toolchain
- **LLM-friendly** - clean, clear patterns for code generation
- **Long-term maintainable** - stable technologies, not experimental
